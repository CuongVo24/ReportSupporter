# Module 3 — Check

> **Vị trí trong stack:** `src/modules/check/` (TechnicalStack §9).
> **Tuần xây dựng:** W3 (checker engine + MVP rules + grouped issues + readiness score draft) — theo `Design/RoadMap/MasterRoadMap.md`.
> **Canonical types:** `ReportIssue`, `ReportProject`, `ReportSection` (Contract W1).
> **HARD RULE:** Checker chạy **OFFLINE** — KHÔNG gọi mạng (VibeCode §4, ProductPRD §6).

---

## 1. Purpose

Check giúp người dùng bắt lỗi báo cáo & lỗi nộp bài **trước khi export**. Nó đọc cấu trúc Markdown, phát hiện section thiếu, evidence thiếu, và các dấu hiệu nội dung yếu/rủi ro, rồi trả về `ReportIssue[]` có severity + vị trí + gợi ý sửa.

Vì sao giữ chân người dùng: nỗi đau lớn nhất khi nộp báo cáo là **nộp xong mới phát hiện thiếu mục lục / thiếu kết luận / quên link demo**. "Check before submit" là một nút bấm cho người dùng cảm giác an toàn — và họ quay lại app mỗi lần sắp nộp. Đây là tính năng tạo thói quen.

Nguyên tắc kỹ thuật: Checker đọc **mdast/hast AST** (heading levels, code lang, image nodes, table width) — KHÔNG regex thô trên string, **trừ** các check văn bản thuần (`TODO`, `lorem ipsum`). Mọi rule phải **deterministic** và phủ test Vitest (TechnicalStack §3, §7).

---

## 2. Scope

### In scope (MVP — W3)

- Phân tích cấu trúc Markdown qua AST.
- Phát hiện các section bắt buộc bị thiếu.
- Phát hiện evidence gap (link demo/source/deploy thiếu cho template phần mềm).
- Phát hiện dấu hiệu nội dung yếu/rủi ro (placeholder text, heading lỗi, ảnh hỏng…).
- Trả `ReportIssue[]` group theo severity, mỗi issue có `suggestion` + `sectionId`/`line`.
- Re-run sau khi sửa và cập nhật kết quả.
- Readiness score draft (W3) — điểm sẵn-sàng-nộp từ tỉ lệ issue.

### Out of scope (theo `ProductPRD.md` §6 Non-goals + VibeCode §4)

- **No network check** — không kiểm link sống bằng HTTP (offline). Chỉ kiểm "có/không có link", không fetch URL.
- **No AI content critique** ở Phase 1 — không chấm văn phong bằng LLM, không "rewrite hộ".
- **No grammar/spell-check engine** ngoài stack — không kéo lib NLP nặng; chỉ rule deterministic.
- **No plagiarism / similarity check** — ngoài scope MVP.
- **No auto-fix** ở MVP — Checker báo + gợi ý, người dùng tự sửa (auto-fix có thể là Phase sau).

## 3. Data Model

### 3.1 Canonical type (giữ NGUYÊN — Contract W1)

```ts
type ReportIssue = {
  id: string;
  severity: "error" | "warning" | "info";
  module: "write" | "format" | "check" | "export";
  message: string;
  suggestion: string;
  sectionId?: string;
  line?: number;
};
```

### 3.2 Module-specific types

```ts
// src/modules/check/types.ts

// Một rule trong engine. Mọi rule implement interface này → dễ test & mở rộng.
type CheckRule = {
  id: string;                 // "missing-toc", "broken-image", ...
  severity: ReportIssue["severity"];
  detect: "ast" | "text";     // đọc AST hay quét text thuần
  // Chạy trên 1 project đã parse, trả về 0..n issue.
  run: (ctx: CheckContext) => ReportIssue[];
};

// Context truyền vào mỗi rule (đã parse sẵn, không parse lại nhiều lần).
type CheckContext = {
  project: ReportProject;
  // mdast của từng section (parse 1 lần, share cho mọi rule).
  sectionAsts: Record<string /*sectionId*/, MdastRoot>;
  // metadata template để biết có phải template phần mềm không.
  templateId: string;
};

// Kết quả tổng hợp khi chạy toàn bộ engine.
type CheckResult = {
  issues: ReportIssue[];
  grouped: {
    error: ReportIssue[];
    warning: ReportIssue[];
    info: ReportIssue[];
  };
  readinessScore: number;     // 0..100 (draft, xem §5.3)
  ranAt: string;              // ISO 8601
};
```

### 3.3 Zod schema sketch

```ts
// src/modules/check/schemas.ts
import { z } from "zod";

export const reportIssueSchema = z.object({

  message: z.string(),
  suggestion: z.string(),
  sectionId: z.string().optional(),

```

## 5. Processing Logic

### 5.1 Engine pipeline

1. Người dùng nhấn "Check before submit" (hoặc auto-run debounce sau edit).
2. Parse **một lần** mỗi `section.markdown` → mdast (remark-parse + remark-gfm + remark-math). Cache vào `CheckContext.sectionAsts`.
3. Lặp qua **registry** các `CheckRule` (mỗi rule đọc AST hoặc text từ context).
4. Gom mọi `ReportIssue` trả về.
5. Group theo `severity` → `grouped`.
6. Tính `readinessScore` (§5.3).
7. Render checker panel; set `ranAt`.
8. **Không gọi mạng** ở bất kỳ bước nào.

### 5.2 Rule catalog (mỗi rule: id · severity · detect · cách phát hiện · suggestion)

| id | severity | detect | Cách phát hiện | Suggestion message |
| :--- | :--- | :--- | :--- | :--- |
| `missing-toc` | warning | ast | Không có heading nào ở depth 1–2, hoặc template yêu cầu TOC mà không có section "Mục lục"/TOC marker. | "Báo cáo chưa có mục lục — bật Table of Contents trong Module Format." |
| `missing-conclusion` | error | ast | Không tìm thấy heading khớp tập từ khoá kết luận ("Kết luận", "Conclusion"). | "Thêm phần Kết luận để tổng kết kết quả & hướng phát triển." |
| `missing-references` | error | ast | Không có heading "Tài liệu tham khảo"/"References". | "Thêm mục Tài liệu tham khảo (References)." |
| `missing-member-table` | warning | ast | Template nhóm/phần mềm nhưng không có `table` mô tả phân công thành viên. | "Thêm bảng phân công nhiệm vụ thành viên." |
| `missing-project-links` | error | ast+meta | Template phần mềm nhưng `metadata`/nội dung thiếu link GitHub/demo/deploy. | "Thêm link source (GitHub), demo, và deploy cho báo cáo phần mềm." |
| `broken-image` | error | ast | `image` node có `url` rỗng, hoặc `asset:<id>` không có trong `project.assets`, hoặc đường dẫn local không hợp lệ. | "Ảnh thiếu hoặc sai đường dẫn — chèn lại ảnh." |
| `heading-jump` | warning | ast | Heading nhảy cấp (vd h1 → h3 bỏ h2) khi duyệt depth tuần tự. | "Heading nhảy cấp — chèn cấp trung gian hoặc hạ cấp heading." |
| `missing-caption` | warning | ast | `image`/`table` node không có caption đi kèm. | "Thêm caption cho hình/bảng để Format đánh số." |
| `table-too-wide` | info | ast | `table` có số cột vượt ngưỡng (vd > 6) → có thể tràn trang A4. | "Bảng nhiều cột có thể tràn trang — cân nhắc tách bảng hoặc xoay landscape." |
| `code-no-lang` | warning | ast | `code` (fenced) node có `lang` null/rỗng. | "Thêm ngôn ngữ cho code block (vd ```ts) để highlight & export đúng." |
| `placeholder-text` | warning | text | Quét text thuần tìm `TODO`, `fix later`, `lorem ipsum` (case-insensitive). | "Còn nội dung placeholder — xoá/thay trước khi nộp." |
| `empty-section` | warning | ast | `section.markdown` rỗng/chỉ whitespace sau khi bỏ heading. | "Section còn trống — viết nội dung hoặc xoá section." |

> **detect = ast** → đọc node mdast/hast (đúng triết lý TechnicalStack §3).
> **detect = text** → chỉ dùng cho check văn bản thuần (`placeholder-text`), được phép regex.

### 5.3 Readiness score (draft — W3)

1. Khởi tạo `score = 100`.
2. Trừ điểm theo severity của mỗi issue: `error` −15, `warning` −5, `info` −1 (clamp ≥ 0).
3. (Tuỳ chọn) cap tối đa số trừ để 1 lỗi lặp nhiều lần không về 0 ngay.
4. Hiển thị badge: ≥85 xanh, 60–84 vàng, <60 đỏ.
5. Score là **draft** — không phải tiêu chí pass/fail tuyệt đối, chỉ định hướng.

## 6. Edge Cases & Smart-Operation Notes ("gia vị vận hành")

- **Empty project (0 section):** Engine chạy không crash, trả issue `empty-section`/`missing-*`, score thấp; không throw.
- **Section rỗng:** `empty-section` flag; rule khác bỏ qua section đó an toàn.
- **Heading toàn h2 (không h1):** `missing-toc`/numbering có thể lệch → vẫn báo, không crash; rule đọc depth thực.
- **Ảnh base64 hợp lệ nhưng to:** `broken-image` không báo (ảnh OK), nhưng Module 1 cảnh báo quota riêng — Check không trùng việc.
- **`asset:<id>` mồ côi:** Ảnh tham chiếu id đã bị xoá khỏi `assets` → `broken-image` error.
- **Link trong text vs trong metadata:** `missing-project-links` kiểm cả `metadata` lẫn `link` node trong AST; có một trong hai là pass — tránh false positive.
- **Placeholder trong code block:** `TODO` trong code mẫu hợp lệ → cân nhắc loại trừ `code` node khỏi `placeholder-text` để giảm nhiễu (hoặc hạ severity về info).
- **Bảng GFM lệch cột:** Bảng cú pháp sai → remark-gfm có thể không parse thành `table` → `table-too-wide` bỏ qua, nhưng preview hiện lỗi; Check không cố sửa.
- **Huge report (nhiều section dài):** Parse 1 lần / section + cache AST trong `CheckContext` → không parse lại mỗi rule (tránh chậm O(rules × parse)).
- **Re-run idempotent:** Chạy lại trên cùng nội dung → cùng `issues` (trừ `ranAt`). Test regression dựa vào tính này.
- **Offline tuyệt đối:** Không có rule nào fetch URL. Kiểm link = kiểm "có chuỗi link", KHÔNG kiểm link sống (Non-goal + offline).
- **Template không phải phần mềm:** `missing-project-links`/`missing-member-table` chỉ áp khi `templateId` thuộc nhóm phần mềm/nhóm → tránh báo sai cho lab report cá nhân.

## 7. Dependencies

### Modules

- **Module 1 (Write):** Nguồn `ReportProject` + `assets` (để check `broken-image`), và là nơi "jump-to-issue" nhảy tới.
- **Module 2 (Format):** Chia sẻ cờ heading-jump / wide-table / missing-caption; Check trình bày chúng thành issue.
- **Module 4 (Export):** Người dùng nên "Check" trước "Export"; export có thể cảnh báo nếu còn `error`.

### Locked stack libs (TechnicalStack.md)

- `unified` + `remark-parse` + `remark-gfm` + `remark-math` — parse AST cho rule (đọc mdast/hast).
- `zod` — validate `ReportIssue` (đặc biệt khi load/lưu).
- `Vitest` — **BẮT BUỘC** test mọi rule (deterministic & regression-safe — TechnicalStack §7).
- **KHÔNG** dùng lib mạng / NLP / spellcheck ngoài stack (offline + Non-goal).

## 9. Acceptance Criteria

- Checker runs **without network access** (offline tuyệt đối).
- Issues grouped by severity (error / warning / info).
- Mỗi issue có `message` + short `suggestion` + `sectionId`/`line` để jump tới.
- User re-run checker sau khi sửa → kết quả cập nhật, re-run idempotent.
- Mọi rule đọc AST (trừ `placeholder-text` quét text) — không regex thô lên cấu trúc.
- Mọi rule có Vitest test (deterministic, regression-safe).
- Rule template-aware: không báo sai cho template không phù hợp (vd link phần mềm với lab report).
- Readiness score (draft) phản ánh số/severity issue và đổi màu theo ngưỡng.
